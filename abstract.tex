\begin{abstract}
  Effect handlers are a programming construct allowing users to define and
  combine computational effects, such as cooperative concurrency. To add
  handlers without compromising safety, one can use a static type-and-effect
  system. The implementation of cooperative concurrency with a higher-order fork
  reveals limitations of such a system.

  We first illustrate these limitations by implementing cooperative concurrency
  in four existing languages. We then present a new language based on modal
  effect types, which we use to implement cooperative concurrency with a
  higher-order fork.
  
\end{abstract}

\section{Introduction}
Effect handlers can express many control-flow constructs. One example is
cooperative concurrency~\cite{structuredasync2017}, which can be modelled using
an effect with two operations \textsf{fork} and \textsf{yield} that,
respectively, create a new task and pause the current one. The languages we will
consider here implement multitasking with a UNIX-style fork of type
$\textsf{unit} \twoheadrightarrow
\textsf{bool}$~\cite{hillerstrom2021thesis}. The continuation receives a boolean
indicating whether it is the parent or child, like in the following piece of
\textsc{Metl} code.
\begin{lstlisting}[style=metl]
  if do fork () then
    print "parent"
  else
    print "child"
\end{lstlisting}
The approach above to multitasking relies on multi-shot continuations, but most
existing languages that start supporting effect handlers only support
single-shot continuations~\cite{ocamleffect2021, wasmeffect2023}.

One can still implement cooperative concurrency with effect handlers in OCaml
using another approach, even though it only supports single-shot
continuation~\cite{dolan2015effective, ocamleffect2021}. The fork operation has
then this signature.
\begin{lstlisting}[style=ocaml]
  type _ Effect.t += Fork : (unit -> unit) -> unit t
\end{lstlisting}
This operation takes as input a thunk of the computation which the new task
should run. The scheduler maintains a queue of thunks that it runs. As OCaml has
no static effect system, the type signatures are straightforward.

We present implementations of a scheduler for cooperative concurrency with
higher-order fork in Koka, Frank, Effekt and \textsc{Metl}. Then, we introduce a
new way to express higher-order effects with modal effect types and use a
language based on it to implement such a scheduler.
\section{Implementation in four languages}\label{impl}
We will now overview implementations of a cooperative concurrency effect with a
fork that takes a thunk as input. To demonstrate compositionality of the
type-and-effect systems, schedulers will use when possible a parametric queue
effect with two operations (\textsf{enqueue} and \textsf{dequeue}) to store
thunks.

All the implementations are available in appendix and
\href{https://github.com/4y8/hope26}{online}.
\subsection{Koka}
Koka has a traditional type-and-effect system where each arrow type holds a
scoped row of effects the function can
perform~\cite{koka2014,scopedrows2005}. The implementation relies on effect
polymorphism to store and yield effectful functions while pleasing the type
system.
\begin{lstlisting}[style=koka]
  div effect co<e>
    ctl yield () : unit
    ctl fork (p : () -> <co<e>,div|e> unit) : unit
\end{lstlisting}
Koka's positivity checker requires the \textsf{div} keyword before the
declaration, and we propagate the \textsf{div} effect, which signals potential
non-termination, through all the signatures. The type of processes is also
effect-polymorphic:
\begin{lstlisting}[style=koka]
  div type proc<e>
    Proc (p : () -> <queue<proc<e>>,div|e> unit)
\end{lstlisting}
With help from the alias \lstinline[style = koka]{qproc e = queue<proc<e>>}, we
implement a scheduler with type:
\begin{lstlisting}[style=koka]
  (() -> <co<e>, qproc<e>, div|e> unit) ->
    <qproc<e>,div|e> unit
\end{lstlisting}
which becomes 
\begin{lstlisting}[style=koka]
  (() -> <co<e>, qproc<e>|e> unit) -> <qproc<e>|e> unit
\end{lstlisting}
if we remove the \textsf{div} effect.
\subsection{Frank}
Noticing that the trailing effect variable is often the same through a
signature, the Frank language gives syntactic sugar to insert
effect variables~\cite{frank2020}.

We transcribe here the concurrency example given
in~\cite[Section~6]{frank2020}. The corresponding effect has signature:
\begin{lstlisting}[style=Frank]
  interface Co = fork : {[Co]Unit} -> Unit
               | yield : Unit
\end{lstlisting}
where \lstinline[style=Frank]!{[Co]Unit}! is the type of computations with
effect \lstinline[style=Frank]{Co}. By defining the type
\lstinline[style=Frank]{Proc} to be \lstinline[style=Frank]!proc {[Queue Proc]Unit}!,
the scheduler has the following variable-free type.
\begin{lstlisting}[style=Frank]
  {<Co>Unit -> [Queue Proc]Unit}
\end{lstlisting}
The type checker desugars this type to:
\begin{lstlisting}[style = Frank]
  {<Co e>Unit -> [Queue (Proc e), e]Unit}
\end{lstlisting}

But Frank's approach is unsatisfactory as it only relies on syntax.
\subsection{\textsc{Metl}}
Drawing on Multimodal Type Theory~\cite{mtt2020}, modal effect types use
\emph{modalities} to give a robust version of Frank's idea of \emph{ambient
ability}, called the \emph{ambient effect context}, and eliminate effect
variables~\cite{tang2025modal, tang2026modal}. To ensure effect safety,
operations can only yield values whose meaning is independent from the ambient
effect context, which is not the case of arrow types, as they may use any effect
in the context. Yielding a function is only possible by first boxing it in an
\emph{absolute} modality that fixes the set of effects it can perform. However,
to be able to compose cooperative concurrency with any effect, we cannot fix a
set of effects and need to bring back effect
variables~\cite[Section~2.10]{tang2025modal}.
\begin{lstlisting}[style=metl]
  eff co (e : Effect)
    = fork : [co e, e](unit -> unit) => unit
    | yield : unit => unit
\end{lstlisting}
We define the polymorphic type of processes like in Koka.
\begin{lstlisting}[style=metl]
  type proc (e : Effect) =
    Proc of [queue (proc e), e](unit -> unit)
\end{lstlisting}
Then, the scheduler has the following type.
\begin{lstlisting}[style=metl]
  forall (e : Effect) . [queue (proc e), e]
    ([co e, queue (proc e), e](unit -> unit) -> unit)
\end{lstlisting}

Because each arrow has an absolute modality, this signature is similar to the
Koka one, and eliminates all the advantages of modal effect
types. Moreover, this implementation relies on a \emph{modality-parametrised
handler}~\cite[Section~3.5]{tang2026modal} to give a type with an absolute
modality to the continuation, bringing extra burden compared to Koka.

\subsection{Effekt}
Effekt takes another path to avoid cluttering type signatures with effect
variables~\cite{effekt2020, effekt2022}. It uses capabilities to track effects,
which gives cleaner type signatures. This approach relies on functions being
second-class citizens in the language.

This particularity offers two ways to define the cooperative concurrency
effect. On the one hand, the \textsf{fork} operation can take a thunk as a
first-class boxed function.
\begin{lstlisting}[style=effekt]
  interface Co[A, B] {
    def yield(): Unit
    def fork(p: Process[A, B]): Unit }

  type Proc[A, B] =
    ({ cap: A => B } => (() => Unit / Co[A, B] at {cap}))
      at {}
\end{lstlisting}
To perform any effect other than concurrency, processes have to use the argument
\texttt{\small cap}.

On the other hand, operations can take as input second-class \emph{blocks}.
\begin{lstlisting}[style = effekt]
  interface Co {
    def yield(): Unit
    def abort(): Nothing
    def forkOp{ p: () => Unit }: Unit }
\end{lstlisting}
When handling \texttt{\small forkOp}, the continuation expects as argument a
function that takes \texttt{\small p} as input. The scheduler resumes the
continuation twice: to enqueue \texttt{\small p} and to continue the
computation, because the type checker does not allow enqueuing \texttt{\small p}
from the continuation's argument. To continue the parent computation only once,
\texttt{\small fork} wraps each fork with an \texttt{\small abort} operation.
\begin{lstlisting}[style = effekt]
  def fork { p: => Unit / Co }: Unit / Co =
    do forkOp { p(); do abort() }
\end{lstlisting}
  
None of the implementations above use an effect for the queue.

The unsatisfactory solution in \textsc{Metl} motivates the search for new ways
to extend the core calculus to support polymorphic higher-order effects. We
have implemented a promising approach: relax kind restrictions on operations’
operands, but restrict subeffecting in presence of non-absolute types. These
conditions should ensure that a non-absolute value arrives to an effect context
more permessive than the one it comes from.

With these new rules, the declaration below is well-typed and models
concurrency.
\begin{lstlisting}[style=metl]
  eff! co = fork : (unit -> unit) => unit
          | yield : unit => unit
\end{lstlisting}
The bang indicates that the effect can yield and receive any value. Unlike in
previous implementations, the signature of \textsf{fork} does not mention the
\textsf{co} effect. In fact, the thunk will have the same ambient effect context
as the computation that yielded it, which contains effect \textsf{co}. For the
same reason, there is no need for a distinct type of processes and can use
functions. The scheduler does not rely modality-paramterised handler and has the
following type.
\begin{lstlisting}[style=metl]
  [queue (unit -> unit)](<co>(unit -> unit) -> unit)
\end{lstlisting}
Another difference with the implementations of Section~\ref{impl} is the absence
of masking. The semantics wraps the operands of higher-order effects with the
same masks as the yielding evaluation context. Therefore, the first call to the
scheduler still masks the \textsf{queue} effect.
\begin{lstlisting}[style=metl]
  val full_schedule : [](<co>(unit -> unit) -> unit)
  let full_schedule m =
    queue { schedule { mask<queue> (m!) } }
\end{lstlisting}

For now, we lack a proof of safety and an expressivity result relating this
system with the one with effect variables.

\subsection{Additional comments}
The version with bidirectional effects resumes the continuation twice. Effekt's
type system reject the following handler for \texttt{\small forkOp}, which would
only require resuming once, because the capability \texttt{\small child} is
incompatible with the queue.
\begin{lstlisting}[style = effekt]
  def forkOp() =
    resume { {child: () => Unit} =>
      queue.pushFront(box child);
      () }
\end{lstlisting}
Such a handler, would remove the need for the wrapper. 

We have not managed to use a queue effect to store our thunks. To do so, we
would need to store in a queue a thunk that captures the queue capabality
itself. However, we could not implement such a queue while staying in Effekt's

\subsection{Theory}
The syntax annotates each effect type with an \emph{accidental}: a flat
signature $A\twoheadrightarrow^\flat B$ has operands of absolute types, and a
sharp signature $A\twoheadrightarrow^\sharp B$ has no restriction. Effect
contexts and extensions are scoped rows of labels that a context map to
signatures.
\begin{center}
\begin{tabular}{ l r c l }
  Accidentals & $Y$ & ::= & $\flat\mid\sharp$\\
  Extensions & $D$ & ::= & $\cdot\mid\ell,D$\\
  Effect Contexts & $E,F$ & ::= & $\cdot\mid\ell,E$\\
  Label Contexts & $\Sigma$ & ::= &
  $\cdot\mid\ell:A\twoheadrightarrow^Y B,\Sigma$\\ 
\end{tabular}
\end{center}
The extension equivalence rules fix the position of sharp signatures. Effect
context equivalence obeys the same rules. All of the following rules suppose a
the existence of global label context $\Sigma$.
\begin{mathpar}
  \inferrule{\\}{\cdot\equiv\cdot}

  \inferrule{D_1\equiv D_2\\ D_2\equiv D_3}{D_1\equiv D_3}

  \inferrule{D_1\equiv D_2}{\ell, D_1\equiv \ell, D_2}

  \inferrule{\ell\neq \ell'\\ \ell:\flat\\ \ell':\flat}
    {\ell, \ell', D \equiv \ell', \ell, D}
\end{mathpar}
Subeffecting of extensions allows moving sharp labels backward.
\begin{mathpar}
  \inferrule{\\}{\cdot\leqslant\cdot}

  \inferrule{D_1\leqslant D_2\\ D_2\leqslant D_3}{D_1\leqslant D_3}

  \inferrule{D_1\leqslant D_2}{\ell, D_1\leqslant \ell, D_2}

  \inferrule{\ell\neq \ell'\\ \ell:\flat}
    {\ell, \ell', D \leqslant \ell', \ell, D}
\end{mathpar}
For subeffecting, effect contexts follow the same rules as extensions with the
addition of $\cdot\leqslant E$.

The rules for $\keyw{do}$ ensure that a sharp operation is unavailable if not
first of the ambient effect context.
\begin{mathpar}
  \inferrule[\textsc{T-DoFlat}]
    {E = D,\ell,F\\
     \ell\not\in D\\
     \Sigma\ni\ell:A\twoheadrightarrow^\flat B\\
     \Gamma\vdash N:A \;@\; E}
    {\Gamma\vdash\Do\ell\; N:B \;@\; E}

  \inferrule[\textsc{T-DoSharp}]
    {\Sigma\ni\ell:A\twoheadrightarrow^\sharp B\\
     \Gamma\vdash N:A \;@\; \ell,E}
    {\Gamma\vdash\Do\ell\; N:B \;@\; \ell,E}
\end{mathpar}

To simplify the rules, we restrict each handler to a single operation. The rule
\textsc{T-HandleSharp} wraps the operands of sharp operations with extension
modalities.
\begin{mathpar}
  \inferrule[\textsc{T-HandleSharp}]
    {\Gamma,\lock_{{\langle|\ell\rangle}_E}\vdash M:A\;@\;\ell,F\\
     \Sigma\ni\ell:A'\twoheadrightarrow^\sharp B'\\
     \Gamma, x:\langle\ell\rangle A\vdash N:B\; @\; F\\
     \Gamma, p:\langle\ell\rangle A', r:\langle\ell\rangle B'\to B\vdash
     N':B\; @\; F
    }
    {\keyw{handle } M\keyw{ with }
     \{\keyw{return }x\mapsto N,\ell\;p\;r\mapsto N'\}
     : B\; @\; F}
\end{mathpar}
Handling flat operation works the same way as usual \textsc{Met}.
